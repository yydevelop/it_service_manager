# FX自動取引プラットフォームの背景（午後Ⅱ論文対応・汎用＋差分設計）

## 2. FX自動取引プラットフォームの背景

本章は、ITサービスマネージャ試験 午後Ⅱにおいて、重大インシデント管理、ITサービス継続性管理、ITサービスの移行、外部委託管理、改善活動など複数の設問に共通して利用できる汎用的な事例背景として整理する。

## 2.1 事業・サービスの位置づけ

- 事業形態  
  - BtoB型ITサービス事業
- 提供サービス  
  - FXブローカー向け自動取引プラットフォーム
- 顧客  
  - 国内FXブローカー（複数社、例：5社）
- エンドユーザー  
  - FXブローカーと直接契約する投資家
- 当社の立場  
  - 投資家とは直接契約・直接サポートを行わない

本サービスはBtoBtoC構造であり、ITサービスマネージャが直接の顧客として対応するのはFXブローカーである。投資家への説明・周知・補償対応は、原則としてFXブローカーが主体となって実施される。

## 2.2 サービス特性（論文での重要ポイント）

- 為替市場に連動し24時間365日稼働するITサービスである
- 東京市場、ロンドン市場、ニューヨーク市場が連続するため、実質的に業務影響なく停止可能な時間帯は存在しない
- 主な処理内容  
  - 注文受付処理  
  - 約定処理  
  - ポジション管理  
  - 取引履歴・残高データの連携
- 特に重要経済指標発表時や要人発言時には、平常時の5倍から10倍の注文が短時間に集中する

障害や性能劣化が発生した場合の影響は以下のとおりである。

- FXブローカーの業務停止および顧客対応停止
- 投資家における取引機会損失や想定外損失
- SNSや比較サイトを通じた評判低下
- 当社およびFXブローカー双方の社会的信用の低下

このため、本サービスでは高い可用性、迅速な復旧、ならびに顧客に対する説明責任が必須となる。

## 2.3 全体の組織構成

- 経営層  
  - CEO（事業責任）  
  - CTO（技術責任）
- 開発課  
  - アプリケーション開発  
  - クラウドインフラ構築・変更
- 運用課（ITサービス提供部門）  
  - ITサービスマネージャ  
  - サービスデスク  
  - 運用担当

ITサービスマネージャは運用課に所属し、開発課および運用担当から独立した立場で、サービス全体の判断・統制・説明責任を担う。

## 2.4 チーム構成（運用課）

- 運用課：計5名  
  - ITサービスマネージャ：1名  
  - サービスデスク担当：2名  
  - 運用担当：2名

24時間365日稼働のITサービスであるが、常時有人対応は採用しない。

- 内製3交代制は、人員規模・コスト・属人化の観点から非現実的と判断した
- 夜間および休日の監視は外部運用監視センター（NOC）に委託する
- 運用課長という別役職は設けず、ITサービスマネージャが運用課を統括する

## 2.5 ITサービスマネージャの所属・役割・責任範囲

### 所属

- 運用課に所属するITサービスマネージャ

### 主な役割

- FXブローカーと締結するSLAおよびSLOの策定および見直し
- 稼働率、障害件数、復旧時間の分析および月次サービス報告
- 重大インシデント発生時の統括  
  - インシデント優先度の判断  
  - サービス再開方針（全面再開・部分再開）の決定  
  - FXブローカーへの説明方針決定
- 変更管理における最終承認
- エラーバジェットを用いた改善判断

### 明確に行わない業務（減点回避）

- 障害対応の実作業
- 監視作業
- シフト管理の実務
- 開発・実装作業

判断・統制・説明責任に専念する立場とする。

## 2.6 サービスデスクの位置づけ

- 運用課に所属
- FXブローカーからの問い合わせ・障害連絡の一次窓口
- チケット起票、分類、優先度設定
- 定型的な顧客連絡および進捗共有
- ITサービスマネージャへの集計および定期報告

## 2.7 運用担当の位置づけ

- 外部NOCからの監視通知内容の確認
- 障害の一次切り分け
- 手順書に基づく暫定復旧
- 開発課へのエスカレーション
- 定常運用作業（ジョブ実行確認、バックアップ確認など）

## 2.8 サービス提供形態（24時間365日対応）

### 基本方針

- 24時間365日稼働 ＝ 常時有人対応ではない
- 内製3交代制は採用しない

### 外部委託（確定）

- 夜間・休日の監視は外部運用監視センター（NOC）に委託
- 外部NOCが24時間365日で監視を実施

### 障害時の対応フロー

- P1（重大障害）  
  - 即時に社内オンコールへ通知  
  - ITサービスマネージャが統括
- P2以下  
  - チケット起票し翌営業日に対応
- 社内オンコール  
  - 運用担当が輪番  
  - ITサービスマネージャはバックアップ対応

## 3. AWS前提の可用性・稼働率設計（汎用）

- 採用クラウド基盤：AWS
- 構成（可用性前提）  
  - EC2（マルチAZ構成）  
  - ALB（マルチAZ）  
  - RDS（Multi-AZ）
- 可用性設計の前提（AWS公開SLAの代表値に整合する設計方針）  
  - EC2：月間99.99%級を前提としたマルチAZ構成  
  - ALB：月間99.99%級を前提としたマルチAZ構成  
  - RDS Multi-AZ：月間99.95%級を前提とした構成

以上を踏まえ、プラットフォーム全体のSLA目標は99.95%と設定する。

- 年間停止許容時間：約4.4時間
- 月間停止許容時間：約22分

稼働率は「インフラが生きているか」だけでなく、「注文受付から約定、取引データ連携までが成立しているか」をサービス視点で定義し、外形監視と内部メトリクスの両方で評価する。

## 4. 午後Ⅱテーマ別 差分（必要な場合のみ使用）

ここは汎用背景を一切変えずに、設問テーマに応じて追加する差分である。差分の追加によって情報量を増やすが、汎用背景の削除・要約は行わない。

### パターンA：重大インシデント管理（例：ピーク負荷・性能劣化）

- 想定事象  
  - 経済指標発表時の注文集中によりDB接続が枯渇し、約定処理が滞留する
- 影響（定量化の例）  
  - 約定不可時間：最大15分  
  - 影響顧客：ブローカー5社中3社へ影響（残り2社は取引量が少なく影響軽微）
- ITサービスマネージャの判断（論文で明示）  
  - 全面再開を急がず、部分再開として「新規注文停止、決済のみ許可」を指示し、投資家の損失拡大を防ぐ
- 効果の例  
  - 完全停止を回避  
  - ブローカーへの一次報告を30分以内に実施し、苦情の急増を抑制する（苦情件数を約40%抑制）

### パターンB：ITサービス継続性管理（例：AZ障害・広域障害）

- 目標（定量）  
  - RTO：30分以内  
  - RPO：0分（取引データ）
- 対策（差分）  
  - DB同期レプリケーション  
  - 復旧手順（Runbook）の整備と更新  
  - 定期的な復旧訓練（年2回、机上訓練と実地訓練）
- 見直しの理由（差分）  
  - 従来想定していなかった広域障害の発生事例を踏まえ、単一リージョン前提の復旧設計を見直す

### パターンC：ITサービスの移行（例：約定エンジン更改）

- 移行対象（差分）  
  - 約定エンジンの更改（機能追加よりも処理安定性・性能改善が目的）
- 制約（定量）  
  - 停止許容時間：最大10分  
  - 切戻し判定：5分以内に実施
- 工夫（差分）  
  - ブルーグリーンデプロイ  
  - 本番相当負荷でのリハーサルを複数回実施し、切替手順の確実性を高める

### パターンD：外部委託管理（例：外部NOC品質・連携）

- 委託範囲（差分）  
  - 監視および一次通知（障害対応の実作業は当社オンコールが実施）
- 合意SLA（定量）  
  - 検知：5分以内  
  - 通知：10分以内
- KPI（差分）  
  - 通知遅延率：1%未満  
  - 誤検知率：月間2%未満  
  - 連絡経路不達：0件を目標
- ITサービスマネージャの統制（差分）  
  - 月次で委託先とレビューを実施し、連絡手順・エスカレーション条件・連絡先一覧を更新する

### パターンE：改善活動（例：SRE・エラーバジェット運用）

- SLO（差分）  
  - 約定成功率：99.99%  
  - P1インシデント：月0件を目標（発生時は原因分析と再発防止を必須）
- エラーバジェット（差分）  
  - 月間0.05%（約22分の停止許容を前提に、改善優先度判断に使用）
- ITサービスマネージャの判断（差分）  
  - エラーバジェット消費が一定量を超えた場合、機能追加リリースを凍結し、安定化改善を優先する
