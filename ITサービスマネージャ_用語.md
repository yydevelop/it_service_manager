## ① 資源管理系
*CPU, ハードディスク, ソウトウェア資源, 構成管理, 変更管理, データ資源管理, ネットワーク資源管理など*

- **資源管理**: システム運用に関する全資源を最高の効率で稼働させ、性能面での過不足がないように調整すること。
- **資源管理の対象**:
  | No | 対象 | 内容 |
  |:--|:--|:--|
  | ① | **ハードウェア** | ホストコンピュータ、サーバ、磁気ディスク装置、磁気テープ装置、パソコン、プリンタ、ワークステーション、ルータ、スイッチングハブなどの機器 |
  | ② | **基本ソフトウェア** | コンピュータを動作させるために必要なOS（オペレーティングシステム）。DBMS、通信制御ソフトウェア、ジョブ制御システムなどの応用ソフトウェアを動作させるために必要なソフトウェアを含める場合も多い |
  | ③ | **応用ソフトウェア** | 販売管理システム、生産管理システム、ワープロ、表計算ソフトなど、ユーザが直接操作するソフトウェア。自社開発する場合と、市販パッケージソフトを購入する場合がある |
  | ④ | **ネットワーク** | インターネット、LAN、広域LAN、VPN、専用回線などの通信経路。必要な通信帯域を確保するための資源 |
  | ⑤ | **データ** | ソフトウェアによって作成された磁気データと紙媒体に記録されたデータ。システム運用上、重視されるのは磁気データ |

- **構成管理 (CMDB)**: サービス要素と関係性を記録するデータベース。影響範囲分析やインシデントの優先度決定に使用する。
- **ゴールデンレコード (Golden Record)**: CMDBにおいて「唯一の正しい情報源（Single Source of Truth）」として認定されたマスターデータ。複数システムに分散した構成情報を統合・名寄せし、重複や不整合を排除した「信頼できる1件」を指す。
  - **目的**: 障害発生時に「どの情報を信じるか」で迷わない。担当者が退職しても属人化しない。
  - **実現方法**: 定期的なディスカバリ（自動検出）ツールによる突合、CI責任者（オーナー）の明確化、変更時のCMDB更新ルール徹底。
  - **論文での使い方**: 「CMDBの情報精度が低く、障害対応が遅延していた」→「ゴールデンレコード運用を導入し、構成情報の一元化と鮮度維持を実現した」という改善ストーリーに使える。
- **キャパシティ管理**: リソース需要を予測し、閾値超過前に増強する。季節変動・キャンペーン負荷を踏まえたモデル提示が効果的。
- **変更管理 (Change Enablement)**: RFCの受付からCAB承認、実施後のレビューまでを統制。緊急変更の例外フローとバックアウト手順の有無を記述できると加点。
- **SBOM (Software Bill of Materials)**: ソフトウェア部品表。アプリに含まれる OSS ライブラリやミドルウェアのバージョン、依存関係を網羅的にリスト化したもの。
  - **具体的使用**: CI/CD パイプラインでビルド時に自動生成（SCAツール利用）し、脆弱性データベース（CVEなど）と照合する。
  - **メリット**: Log4jのような広範囲に影響する脆弱性が公開された際、自社の数あるサービスのうち「どこに該当ライブラリが使われているか」を数分で特定でき、初動対応を劇的に早められる。
- **FinOps (Financial Operations)**: クラウドコストの最適化を、財務・開発・運用の3者が連携して継続的に行う取り組みや文化。「抑制（ケチる）」ではなく、ビジネス価値に対する「投資対効果の最大化」を目指す点が特徴。
  - **Unit Economics (ユニットエコノミクス)**: 1顧客あたり、あるいは1トランザクション（取引）あたりのコストを指標化し、「売上の伸びに伴って適切にコストが増えているか（健全な赤字か）」を判断する。単なる総額管理では見えない「利益率の悪化」を検知するために不可欠。
  - **Showback / Chargeback**: コストが発生した部門やプロジェクトに費用を可視化（Showback）または実際に請求（Chargeback）し、開発チームにコスト意識（オーナーシップ）を持たせる文化醸成の手法。タグ付け（Tagging）によるリソースの紐付けが前提となる。

## ② インシデント管理系
*障害発生時の応急措置, 原因究明, 再発防止策など*

- **インシデント**: 予定外のサービス中断や品質低下。目標は速やかな復旧。一次対応の手順化（SOP）とナレッジ検索性が得点源。
- **問題 (Problem)**: 繰り返すインシデントの根本原因。恒久対策（KEDB/RCFA）で再発防止を狙う。午後Ⅰの設問では「リアクティブ→プロアクティブ」の転換がよく問われる。
- **自動復旧 (Self-Healing)**: 障害検知時に、人が介在せずにシステムが自律的に回復動作を行う仕組み。
  - **例**: オートスケーリング（負荷増時の自動増強）、再起動（ヘルスチェック失敗時のコンテナ再作成）、自動フェイルオーバー（DB障害時のスタンバイ機昇格）。
- **監査ログ管理**: システム操作やイベントの記録を確実に保全する活動。
    - **目的**: 不正アクセスの追跡（フォレンジック）、内部統制の証明（いつ誰が何をしたか）。
    - **ポイント**: ログの「対象（認証、重要データ操作）」、「保管期間（法令や内規に基づく）」、「改ざん防止（WORMストレージやハッシュチェーン）」が論点になる。

## ③ パフォーマンス管理系
*性能向上, コスト削減, 差異分析など*

- **性能管理 (レスポンス管理)**: APMでボトルネックを特定し、スロークエリやGC設定を改善。ピーク時P95応答時間を指標にすると論述しやすい。
- **パーセンタイル (Percentile)**: データを小さい順に並べたとき、何%目の位置にあるかを示す指標。「平均値」は一部の極端に遅いデータ（外れ値）に引っ張られ、実態を見誤るリスクがあるため、SLAではパーセンタイルを用いるのが標準。
    - **P50 (中央値)**: 全体のど真ん中のユーザーの体感速度。
    - **P95 / P99**: 「最も遅い部類」のユーザー体験。例えばP99が3秒なら、「99%のユーザーは3秒以内だが、残り1%（100人に1人）は3秒以上待たされている」ことを意味する。ITSMではこの「不運な1%」を放置しないためにP95/P99を管理指標にする。
    - **IPA公式の言及**:
        - **統計指標に基づく品質マネジメント実践集**: 工数見積もりの妥当性評価において、「25パーセンタイル値〜75パーセンタイル値の範囲」という形で使用されている。
        - **非機能要求グレード**: 性能目標値として「レスポンスタイムの95%順守」などが定義されており、これは実質的にパーセンタイル（P95）と同義である。
- **クラウド時代の可用性 (Composite Availability)**:
  - **直列稼働率**: クラウドサービス(SLA 99.9%)を10個組み合わせると、システム全体の理論稼働率は約99.0% ($0.999^{10}$) に低下する。「クラウドだから高可用性」ではなく、Multi-AZ活用(99.99%)が必須である根拠に使う。
  - **隠れダウンタイム (Micro-stalls)**: クラウドベンダーのSLA定義では「1分未満の停止（ストール）」は稼働とみなされる場合がある。これを「実質的な障害」として検知・切り離すリトライ設計が重要。

## ④ セキュリティ管理系
*不正アクセス防止, ID管理, 啓蒙活動など*

- **ISMS / ISO 27001**: 情報セキュリティマネジメントの枠組み。単なるルール作りではなく、「情報資産の洗い出し→リスクアセスメント（機密性・完全性・可用性）→リスク対応（受容・回避・転嫁・低減）→教育・監査」というPDCAサイクルを回し続けるプロセスそのものを指す。午後問題では、環境変化（テレワーク導入、クラウド移行）に伴うリスク再評価の文脈で頻出。
- **アクセス管理**: ユーザーに対し、適切な権限を付与し、不正なアクセスを防ぐプロセス。
  - **RBAC**: 個人ごとではなく「役割（ロール）」に権限を割り当てることで管理負荷を下げる。
  - **特権ID管理**: システム管理者などの強力な権限を持つIDは、貸出制（ワークフロー申請）、操作ログの常時取得、パスワードの都度変更（ワンタイムパスワード）などで厳重に管理する。内部不正対策の要。
- **IDaaS (Identity as a Service)**: クラウド型のID管理・認証基盤サービス。Auth0, Okta, Microsoft Entra ID（旧Azure AD）などが代表例。
  - **メリット**: 多要素認証（MFA）、リスクベース認証、SSO（シングルサインオン）などの高度な認証機能を、自社開発なしで即座に導入可能。認証基盤の維持管理（トイル）を外部に委託することで、開発リソースを本業に集中できる。
  - **OpenID Connect (OIDC)**: IDaaS連携で使われる標準認証プロトコル。アプリ側は認証処理を一切持たず、IDaaSへリダイレクトするだけで認証を完結できる。
- **パスキー (Passkey) / FIDO2認証**: パスワードに依存しない次世代認証技術。生体認証（指紋/顔）とデバイス内の秘密鍵を用いる。
  - **フィッシング耐性**: 認証情報が正規ドメイン（オリジン）にのみ紐づくため、偽サイトでは認証動作が技術的に不可能。「パスワードがある限りフィッシングは防げない」という課題を根本解決する。
  - **UX向上**: ユーザーはパスワードを覚える必要がなく、ログイン成功率が向上。パスワードリセット依頼によるサービスデスク負荷も激減する。
  - **関連用語**: WebAuthn（ブラウザAPI）、FIDO2（国際標準規格）、オリジンバインディング（ドメイン紐付け）、Attestation（デバイス真正性証明）。

## ⑤ リリース管理系
*移行テスト, 移行準備, リハーサル, 本番切替えなど*

- **リリース管理**: 本番反映の計画と実行。フリーズ期間・リードタイム・デプロイ自動化（CI/CD）などを絡めると具体性が増す。
- **ドライラン (Dry Run) / リハーサル**: 本番環境での作業や重大な変更を行う前に、テスト環境などで本番と全く同じ手順を模擬的に実行すること。「空運転」「予行演習」とも呼ばれる。
  - **目的**: 手順書の不備、権限不足、予期せぬエラー、所要時間の見積もり誤差を事前に洗い出し、本番作業の失敗リスクを最小化する。特に不可逆的なデータ移行や大規模リリース前には必須とされる。
  - **IPAでの位置づけ**: ITサービスマネージャ試験やシステム移行のガイドライン等において、移行テストや運用受入テストの一部（移行リハーサル）、または緊急時の予行演習として、その実施が強く推奨されている。
- **CAB (Change Advisory Board)**: 変更の影響・リスクを評価し承認する委員会。緊急変更の簡易CABや事後レビューも触れると網羅性が出る。

## ⑥ サービスレベル管理系
*サービス基準の設定, 維持, 改善, 顧客満足度向上など*

- **SLA (Service Level Agreement)**: 利用者と提供者で合意するサービス品質指標の集合。可用性99.9%や応答時間3秒以内など数値で定義し、違反時の責任や是正手段も明文化する。
    - **SLA/非機能要求グレード早見表（可用性の目安）**:
        | 稼働率 | 停止時間（年/月） | IPA非機能 | 適用システムの目安 |
        | :--- | :--- | :--- | :--- |
        | **99.999%** | 年5分 / 月25秒 | レベル5 | 決済・金融（無停止必須）、人命に関わるシステム |
        | **99.99%** | 年52分 / 月4分 | レベル4 | 24H365日停止不可のWebサービス（FX、大規模EC） |
        | **99.9%** | 年8.7時間 / 月43分 | レベル3 | 一般的な重要業務システム（日中稼働、夜間計画停止あり） |
        | **99.5%** | 年43.8時間 / 月3.6時間 | レベル2 | 社内システム、一部のSaaS（日曜深夜に計画メンテ許容） |
        | **99.0%** | 年3.6日 / 月7.2時間 | - | 初期段階のWebサービス、利用頻度の低い社内ツール |
- **SLM (Service Level Management)**: SLA/SLOの策定・監視・見直しを行うプロセス。KPIを閾値でモニタし、乖離時にCAPA（是正・予防処置）を起動する。
- **SLA / SLO / SLI の階層構造（User Happinessの防衛）**:
  - **SLI (Indicator = 測定器)**: 現状の数値を測る指標。例：可用性99.2%、応答時間180ms。
  - **SLO (Objective = 内部目標)**: ユーザーの「安心・満足（User Happiness）」を守るための目標値。SLAより厳しく設定する（SLO > SLA）。このギャップが「エラーバジェット」となる。
  - **SLA (Agreement = 対外的な約束)**: 契約上の最低保証値。これを割ると返金やペナルティが発生する「ユーザーが激怒するライン」。
  - **覚え方**: 「測る（SLI）≦ 目標（SLO）≦ 約束（SLA）」
- **エラーバジェット**: 1-SLOで許容されるダウンタイムの上限。消費率が高いほどリリースを抑制し、信頼性改善を優先する判断根拠になる。
- **可用性管理**: 単一障害点の排除、冗長構成、フェイルオーバ試験。MTBF/MTTRを提示し、改善効果を数値で書けるようにする。

## ⑦ サービスデスク系
*相談窓口, FAQなど*

- **サービスデスク**: 単一窓口（SPOC）。一次解決率、平均応答時間、エスカレーション率などを管理指標として記述すると実務感が出る。
- **ナレッジ管理 (KEDB/FAQ)**: 過去インシデントの原因・回避策を蓄積し、検索性を高める。検索成功率や一次解決率向上を効果として書く。
- **WFM (Work Force Management) システム**: 要員管理システム。サービスデスクの入電予測と要員配置を最適化する。
  - **詳細**: 過去の問合せパターンから時間帯別需要を予測し、スキル要件と稼働可能人員をマッチングして勤務シフトを自動生成する。応答率（ASA/AHT）を維持しつつ人件費を最小化するための計画立案と実績トラッキングを行う。要員不足時のアラート発報や、リアルタイムでのシフト調整支援も含む。サービスデスクの効率化と品質維持の両立を実現する際の論述に有効。

## ⑧ その他全般管理
*運用方式設計, 運用体制, 改善活動, 新技術採用など*

- **ITIL v4 プラクティス**: 価値共創やバリューストリームを重視した現代的なベストプラクティス集。
- **CSI (Continual Service Improvement)**: 継続的サービス改善の7ステップ（測定→分析→改善）。KPI→ギャップ分析→改善ロードマップ→効果測定の流れを押さえる。
- **BCP / ITSCM / DR**: 事業継続計画とITサービス継続性管理。RTO（いつまでに）、RPO（いつのデータまで）を具体的に定める。
- **SRE (Site Reliability Engineering)**: Googleが提唱した、ソフトウェア開発の技法を運用に持ち込むアプローチ。「運用担当者」ではなく「エンジニア」として、トイル（手作業による反復業務）をコード化・自動化によって撲滅する。
- **SLIモニタリング基盤**: 可観測性（Observability）の整備。従来の「死活監視」だけでなく、「ユーザが体験している遅延（Latency）」や「エラー率」など、サービスの健康状態をリアルタイムに可視化するダッシュボード。Prometheus, Datadog, Grafana 等が代表例。
- **リスク管理**:
  - **リスク評価**: 発生確率×影響度で優先度付けし、回避・低減・受容・転嫁の方針を決定する。
  - **リスクマトリックス**: 発生確率と影響度を軸にリスクを視覚化する表。縦軸に発生確率（高・中・低）、横軸に影響度（大・中・小）を配置し、リスクを色分けして優先順位を判断する。赤色は即時対応、黄色は監視、緑色は受容など、リスク対応方針の意思決定を支援する。午後Ⅱで「リスクマトリックスで可視化して優先度を決定した」と記述すると効果的。
- **ステークホルダ管理**: 利用部門・経営層・ベンダの利害を調整。合意形成プロセスを示すと午後Ⅱの説得力が増す。
- **エラープルーフ化 (Poka-Yoke)**: エラーを防止するための考え方で、ミスを発生させないように、または発生した場合の影響を最小限に抑えるための対策を講じるもの。五つの原理がある。
  - **排除**: 作業や注意を不要にする
  - **代替化**: 人が作業しなくてもよいようにする
  - **容易化**: 作業を易しくする
  - **異常検知**: エラーに気付くようにする
  - **影響緩和**: 影響が致命的なものにならないようにする
  
  排除・代替化・容易化は作業ミスを未然に防ぐもので、異常検知・影響緩和は発生した作業ミスを検知・軽減するもの。運用ミス由来のインシデント削減の文脈で有効。
